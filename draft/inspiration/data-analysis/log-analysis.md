# 日志分析

## 干什么
自动分析系统日志，提取错误信息、异常模式、性能瓶颈，生成运维报告。

## 怎么干

### 1. 准备工作
- 需要 OpenClaw 版本：>= 1.0.0
- 需要的技能：无特殊要求
- 需要的配置：日志文件访问权限

### 2. 执行步骤

**方式一：分析本地日志**

```
分析这个日志文件：
~/logs/app-2026-03-24.log

重点：
1. 统计错误类型和数量
2. 找出高频错误
3. 分析错误发生时间分布
4. 提取错误上下文
```

**方式二：分析 Nginx 访问日志**

```
分析 Nginx 访问日志：
/var/log/nginx/access.log

输出：
- PV/UV 统计
- Top 10 访问路径
- 状态码分布
- 慢请求（> 1s）列表
- 可疑 IP（高频访问）
```

**方式三：实时监控**

```
持续监控日志文件：
~/logs/app.log

当出现以下关键词时通知我：
- ERROR
- Exception
- timeout
- 500
```

**方式四：日志对比**

```
对比两个时间段的日志差异：
- 文件1：app-2026-03-23.log
- 文件2：app-2026-03-24.log

分析：
- 新增的错误类型
- 错误频率变化
- 可能的关联原因
```

**OpenClaw 会自动：**
1. 解析日志格式
2. 提取关键字段
3. 统计分析
4. 生成可视化报告

### 3. 效果展示

```markdown
# 日志分析报告 | app-2026-03-24.log

## 概览
- 总日志行数：125,678
- 时间范围：00:00:00 - 23:59:59
- 错误数量：1,234 (0.98%)
- 警告数量：3,456 (2.75%)

## 错误统计

| 错误类型 | 数量 | 占比 | 示例 |
|----------|------|------|------|
| NullPointerException | 456 | 37.0% | 用户信息为空 |
| ConnectionTimeout | 234 | 19.0% | 数据库连接超时 |
| SQLException | 189 | 15.3% | 查询超时 |
| IOException | 167 | 13.5% | 文件读取失败 |

## 时间分布
```
错误高峰时段：
- 10:00-11:00 (156 条) - 早高峰流量
- 14:00-15:00 (134 条) - 数据库备份
- 20:00-21:00 (189 条) - 晚高峰流量
```

## 高频错误详情

### 1. NullPointerException (456 条)
- 发生位置：UserService.getUser()
- 触发条件：用户 Token 过期但未正确处理
- 建议：增加 Token 校验逻辑

### 2. ConnectionTimeout (234 条)
- 发生位置：DatabasePool.getConnection()
- 触发条件：数据库连接池耗尽
- 建议：增加连接池大小或优化查询

## 异常模式发现
- 10:15 和 10:45 出现批量超时，可能与定时任务有关
- 用户 ID 以 88 开头的请求失败率较高，疑似测试账号

## 建议措施
1. 修复 Token 校验逻辑
2. 优化数据库连接池配置
3. 排查定时任务对系统的影响
4. 清理或隔离测试账号
```
